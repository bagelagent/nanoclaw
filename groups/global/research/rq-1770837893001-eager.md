# Claude SDK eager_input_streaming Flag Research

**Research ID:** rq-1770837893001-eager
**Topic:** Claude SDK eager_input_streaming flag - how does it affect tool parameter streaming granularity?
**Research Date:** 2026-02-26
**Status:** Complete

---

## Summary

The `eager_input_streaming` flag is a feature in Claude's API that enables **fine-grained tool streaming** - the ability to stream tool parameter values to the client incrementally, without buffering for JSON validation. This dramatically reduces latency (from ~15s to ~3s in documented examples) when Claude generates large tool parameters, at the cost of potentially receiving incomplete or invalid JSON that requires additional error handling.

---

## Key Findings

### 1. What is eager_input_streaming?

`eager_input_streaming` is a **per-tool configuration flag** that controls whether Claude should:
- **Default behavior (false):** Buffer tool parameters until they form valid JSON before streaming to the client
- **Eager streaming (true):** Stream parameter chunks immediately as they're generated, without JSON validation

**Configuration:**
```json
{
  "name": "make_file",
  "description": "Write text to a file",
  "eager_input_streaming": true,  // ← Enable fine-grained streaming
  "input_schema": {
    "type": "object",
    "properties": {
      "filename": { "type": "string" },
      "lines_of_text": { "type": "array" }
    }
  }
}
```

### 2. Impact on Streaming Granularity

The flag fundamentally changes the chunking behavior of tool parameter streams:

**Without eager_input_streaming (default):**
- 15-second delay before first chunk arrives
- Small, frequent chunks with many word breaks
- Each chunk is a valid partial JSON fragment
- Parameters are buffered and validated before streaming

```
Chunk 1 (15s delay): '{"'
Chunk 2: 'query": "Ty'
Chunk 3: 'peScri'
Chunk 4: 'pt 5.0 5.1 '
Chunk 5: '5.2 5'
Chunk 6: '.3'
Chunk 8: ' new f'
Chunk 9: 'eatur'
...
```

**With eager_input_streaming enabled:**
- 3-second delay before first chunk (5x faster time-to-first-byte)
- Larger, less frequent chunks with fewer word breaks
- Chunks may be invalid or incomplete JSON
- No buffering or validation—raw generation output

```
Chunk 1 (3s delay): '{"query": "TypeScript 5.0 5.1 5.2 5.3'
Chunk 2: ' new features comparison'
```

### 3. Performance Implications

**Latency Reduction:**
- **Time-to-first-byte:** ~80% reduction (15s → 3s in examples)
- **Perceived responsiveness:** Significant improvement for UX
- **Throughput:** Eliminates buffering overhead for large parameters

**Use Cases Best Suited for eager_input_streaming:**
- Tools with array parameters (lists, lines of text)
- Tools accepting large text blocks
- Streaming content creation (writing files, generating documents)
- Real-time display of generated content
- Interactive applications where perceived latency matters

**Use Cases Where Default is Better:**
- Small, structured parameters (IDs, flags, enums)
- Complex nested objects requiring validation
- Tools where invalid JSON would cause errors
- Stateless environments that can't handle incremental parsing

### 4. Error Handling Requirements

Enabling `eager_input_streaming` introduces **three critical failure modes:**

**A. Incomplete JSON from max_tokens limit:**
```json
{
  "lines": ["Line 1", "Line 2", "Line 3", "Line
  // Stream ended mid-parameter
```

**B. Invalid JSON syntax:**
```json
{
  "text": "Some content with unescaped "quotes"
  // Missing closing brace, invalid escaping
```

**C. Partial parameter values:**
```json
{
  "filename": "report.txt",
  "content": ["First line", "Sec
  // Array element cut off
```

**Recommended Error Handling Pattern:**

When receiving invalid JSON in a tool response, wrap it for the model:
```json
{
  "INVALID_JSON": "<escaped invalid json string>"
}
```

This preserves the malformed data for debugging while signaling to Claude that validation failed.

**Client Implementation Requirements:**
- Stateful, incremental JSON parsing
- Buffer management for incomplete chunks
- Graceful degradation when max_tokens reached
- Retry logic or fallback handling
- UI indication of streaming vs. complete parameters

### 6. Relationship to Agent SDK Streaming Modes

The `eager_input_streaming` flag is **orthogonal** to the Agent SDK's input modes:

**Agent SDK Input Modes:**
- **Streaming Input Mode:** User messages stream in via AsyncGenerator
- **Single Message Input:** One-shot queries with optional session resumption

**Tool Parameter Streaming (eager_input_streaming):**
- Controls how **Claude's tool outputs** are streamed to the client
- Independent of how user input is provided to Claude
- Affects `content_block_delta` events with `input_json_delta` type

**Both can be used together:**
```typescript
// Streaming input mode + eager tool parameter streaming
async function* generateMessages() {
  yield { type: "user", message: { role: "user", content: "Write a long document" }};
}

for await (const message of query({
  prompt: generateMessages(),
  options: {
    tools: [{
      name: "write_file",
      eager_input_streaming: true,  // ← Tool param streaming
      // ...
    }]
  }
})) {
  // Receive eager-streamed tool parameters
}
```

### 7. Technical Implementation Details

**Event Stream Structure:**

Without eager streaming:
```
event: content_block_delta
data: {"type": "content_block_delta", "index": 1, "delta": {"type": "input_json_delta", "partial_json": "{\"location\": \"San Fra"}}

event: content_block_delta
data: {"type": "content_block_delta", "index": 1, "delta": {"type": "input_json_delta", "partial_json": "ncisco\""}}
```

With eager streaming:
```
event: content_block_delta
data: {"type": "content_block_delta", "index": 1, "delta": {"type": "input_json_delta", "partial_json": "{\"location\": \"San Francisco\", \"activity\":"}}
```

**Chunking Behavior:**
- Current models emit one complete key-value property at a time
- Delays occur between events while the model generates
- Events are chunked into multiple partial JSON deltas
- Future models may provide finer granularity

## Deep Dive: When to Use eager_input_streaming

### Decision Framework

**Use eager_input_streaming = true when:**
1. Tool parameters are **large** (>1KB typical)
2. Parameters are **array-like or streaming-friendly** (lines, items, chunks)
3. User experience benefits from **perceived responsiveness**
4. Application can handle **incremental parsing**
5. Max tokens limit is unlikely to be hit mid-parameter
6. Development team has capacity to implement robust error handling

**Use eager_input_streaming = false (default) when:**
1. Parameters are **small and structured** (<100 bytes)
2. Application expects **valid JSON only**
3. Running in **stateless environment** (lambda, serverless)
4. Tool logic cannot handle invalid JSON gracefully
5. Simplicity > latency optimization
6. No incremental parsing infrastructure exists

### Example Scenarios

**✅ Good Use Case: File Creation Tool**
```json
{
  "name": "create_document",
  "eager_input_streaming": true,
  "input_schema": {
    "properties": {
      "title": { "type": "string" },
      "paragraphs": { "type": "array", "items": { "type": "string" } }
    }
  }
}
```
- Large array parameter ideal for streaming
- User sees paragraphs appear in real-time
- Partial content still useful if max_tokens hit

**❌ Poor Use Case: Database Query Tool**
```json
{
  "name": "run_query",
  "eager_input_streaming": false,  // Keep default
  "input_schema": {
    "properties": {
      "table": { "type": "string" },
      "conditions": {
        "type": "object",
        "properties": { "id": { "type": "integer" }, "status": { "type": "string" } }
      }
    }
  }
}
```
- Small, structured parameters
- Invalid JSON would cause query errors
- No UX benefit from streaming (query can't execute until complete)
- Validation important for security (SQL injection prevention)

## Connections to Existing Knowledge

### Relationship to Streaming Architectures

`eager_input_streaming` aligns with broader trends in streaming systems:

1. **Time-to-First-Byte (TTFB) Optimization:**
   - Similar to HTTP streaming, CDN edge optimization
   - Trade buffering/validation for perceived speed
   - User studies show <3s response time critical for engagement

2. **Incremental Parsing:**
   - JSON streaming parsers (e.g., json-stream Python library)
   - Event-based parsing (SAX for XML, streaming JSON)
   - Stateful buffer management required

3. **Backpressure and Flow Control:**
   - Client must handle fast streaming without blocking
   - Need buffering strategies for slow downstream consumers
   - Consider rate limiting if writing to disk/database

### Comparison to Other AI Streaming Features

**OpenAI Function Calling:**
- Does not have equivalent to `eager_input_streaming`
- Always buffers function arguments until valid JSON
- Higher latency but simpler client implementation

**Anthropic Extended Thinking (Claude Opus 4.0+):**
- Similar streaming of internal reasoning
- Also requires handling incomplete thoughts
- Different use case (transparency vs. performance)

**Google Gemini Function Calling:**
- Validates parameters before streaming
- More conservative approach
- Better for production reliability
