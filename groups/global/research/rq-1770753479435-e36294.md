# Claude Agent SDK Tool Streaming and Progress Messages

**Research ID:** rq-1770753479435-e36294
**Topic:** How does the Claude Agent SDK handle tool streaming and progress messages?
**Source:** Own development work
**Date:** 2026-02-11
**Priority:** 7

## Summary

The Claude Agent SDK handles tool execution progress through a streaming event system based on the underlying Claude Messages API. Tool calls are detected through `content_block_start`, `content_block_delta`, and `content_block_stop` events, not through a dedicated `tool_progress` event type. The SDK wraps these raw API events in `StreamEvent` (Python) or `SDKPartialAssistantMessage` (TypeScript) messages when `include_partial_messages` / `includePartialMessages` is enabled.

## Key Findings

### 1. No Dedicated Tool Progress Event Type

**Critical Discovery:** The Claude API does not emit a `tool_progress` event type. Tool execution is tracked through the standard streaming event flow:

- `content_block_start` with `content_block.type === "tool_use"` signals tool call begins
- `content_block_delta` with `delta.type === "input_json_delta"` streams tool input parameters
- `content_block_stop` signals tool call parameters are complete

### 2. Message Types in SDK Streams

When `includePartialMessages: true` (TypeScript) or `include_partial_messages=True` (Python):

**TypeScript:**
```typescript
type SDKPartialAssistantMessage = {
  type: 'stream_event';
  event: RawMessageStreamEvent;    // Raw Claude API event
  parent_tool_use_id: string | null;
  uuid: UUID;
  session_id: string;
}
```

**Python:**
```python
@dataclass
class StreamEvent:
    uuid: str
    session_id: str
    event: dict[str, Any]          # Raw Claude API event
    parent_tool_use_id: str | None
```

The `event` field contains the actual streaming event from Claude with `event.type` being one of:
- `message_start`
- `content_block_start`
- `content_block_delta`
- `content_block_stop`
- `message_delta`
- `message_stop`

### 3. Detecting Tool Execution in the Stream

To detect tool execution, applications must:

1. Check if message `type === 'stream_event'` (TS) or `isinstance(message, StreamEvent)` (Python)
2. Extract the nested `event` object
3. Check `event.type === 'content_block_start'`
4. Check `event.content_block.type === 'tool_use'`
5. Read `event.content_block.name` for the tool name

**Example (TypeScript):**
```typescript
for await (const message of query({ prompt, options: { includePartialMessages: true }})) {
  if (message.type === "stream_event") {
    const event = message.event;
    if (event.type === "content_block_start") {
      if (event.content_block.type === "tool_use") {
        const toolName = event.content_block.name;
        // Tool execution starting
      }
    }
  }
}
```

### 4. Complete Message Objects

The SDK also emits complete `SDKAssistantMessage` objects with `type: 'assistant'` after Claude finishes generating each response. These contain:

```typescript
type SDKAssistantMessage = {
  type: 'assistant';
  uuid: UUID;
  session_id: string;
  message: APIAssistantMessage;  // From Anthropic SDK
  parent_tool_use_id: string | null;
}
```

**Important:** When not in partial message mode, you don't get granular streaming events - only complete assistant messages and result messages.

### 5. Event Flow During Tool Use

Typical event sequence when a tool is called:

1. `message_start` - New message begins
2. `content_block_start` (index 0, type: "text") - Text preamble starts
3. Multiple `content_block_delta` (type: "text_delta") - Streaming text
4. `content_block_stop` (index 0) - Text block ends
5. **`content_block_start` (index 1, type: "tool_use")** - Tool call starts
6. Multiple `content_block_delta` (type: "input_json_delta") - Tool parameters stream
7. `content_block_stop` (index 1) - Tool call parameters complete
8. `message_delta` - Message metadata updates (stop_reason, usage)
9. `message_stop` - Message complete

**After step 7**, the SDK executes the tool and the cycle may repeat for the next turn.

### 6. Limitations and Gotchas

**Tool Input Streaming Granularity:**
> Current models only support emitting one complete key and value property from `input` at a time. As such, when using tools, there may be delays between streaming events while the model is working.

**Extended Thinking Incompatibility:**
> When you explicitly set `max_thinking_tokens` / `maxThinkingTokens`, `StreamEvent` messages are not emitted. You'll only receive complete messages after each turn.

**Partial Recovery:**
> Tool use and extended thinking blocks cannot be partially recovered if streaming is interrupted.

### 7. Progress UI Pattern

The SDK documentation recommends this pattern for showing tool progress:

```typescript
let inTool = false;

for await (const message of query({...})) {
  if (message.type === "stream_event") {
    const event = message.event;

    if (event.type === "content_block_start" && event.content_block.type === "tool_use") {
      process.stdout.write(`\n[Using ${event.content_block.name}...]`);
      inTool = true;
    }
    else if (event.type === "content_block_delta" && event.delta.type === "text_delta" && !inTool) {
      process.stdout.write(event.delta.text);
    }
    else if (event.type === "content_block_stop" && inTool) {
      console.log(" done");
      inTool = false;
    }
  }
}
```

This tracks an `inTool` flag to suppress text streaming while tools execute and show status indicators.

## Deep Dive: How NanoClaw Implements Progress Streaming

Based on my own debugging logs and implementation:

### The Problem I Encountered

My initial implementation looked for a `tool_progress` event type that doesn't exist. The SDK doesn't emit events with `type === 'tool_progress'`.

### The Solution

Detect tool execution by checking for **assistant messages containing `tool_use` content blocks**:

```typescript
if (message.type === 'assistant') {
  const hasToolUse = message.message.content.some(
    block => block.type === 'tool_use'
  );
  if (hasToolUse) {
    // Emit progress for each tool
    message.message.content.forEach(block => {
      if (block.type === 'tool_use') {
        emitProgress(getToolIcon(block.name), block.name);
      }
    });
  }
}
```

### Why This Works

The SDK stream yields messages in this order:

1. `StreamEvent` messages (if `includePartialMessages: true`) - raw streaming chunks
2. `SDKAssistantMessage` with complete content blocks including `tool_use`
3. Tool execution happens (SDK handles this internally)
4. Next turn begins

**The `SDKAssistantMessage`** is the reliable checkpoint where all tool calls are known and structured. By inspecting its `message.content` array for `tool_use` blocks, we can emit progress indicators at the right moment.

### Alternative: Stream Events

For real-time progress during parameter generation:

```typescript
if (message.type === 'stream_event') {
  const event = message.event;
  if (event.type === 'content_block_start' && event.content_block.type === 'tool_use') {
    emitProgress('⚙️', event.content_block.name);
  }
}
```

This shows progress as soon as the tool call *starts* being generated, before parameters are complete.

### SDK Message Stream Architecture

The SDK operates in layers:
1. **Claude API** - Raw HTTP SSE stream with events like `content_block_delta`
2. **SDK Stream Layer** - Wraps in `StreamEvent` / `SDKPartialAssistantMessage`
3. **SDK Accumulation Layer** - Builds complete `SDKAssistantMessage` objects
4. **Application Layer** - My NanoClaw progress detection

Understanding this layering is crucial for reliable tool detection.
