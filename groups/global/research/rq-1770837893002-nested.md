## Summary

The `parent_tool_use_id` field in the Claude Agent SDK serves as a correlation mechanism for tracking subagent execution and preventing infinite loops in hook systems. When a parent agent spawns a subagent via the Task tool, messages from within that subagent's context include `parent_tool_use_id` pointing back to the originating Task tool call. This enables progress tracking across nested agent operations and provides critical safety mechanisms for hook-based automation.

**Key limitation:** Subagents cannot spawn other subagents—the SDK intentionally prevents nested subagent spawning to avoid infinite recursion.

## Key Findings

### 1. What is `parent_tool_use_id`?

The `parent_tool_use_id` is a field present in SDK messages that identifies whether a message originates from within a subagent's execution context. Its primary purposes are:

1. **Correlation:** Links subagent messages back to the parent Task tool call that spawned them
2. **Context detection:** Allows hooks to determine if they're running in a subagent context
3. **Loop prevention:** Enables hooks to avoid spawning additional subagents when already inside one
4. **Progress tracking:** Helps monitor subagent completion via `SubagentStop` hooks

### 2. Message Structure with `parent_tool_use_id`

When processing streamed messages, `parent_tool_use_id` appears in messages that occur within a subagent's execution:

**Python SDK:**
```python
if hasattr(message, 'parent_tool_use_id') and message.parent_tool_use_id:
    print("This message is from within a subagent")
    print(f"Parent tool use ID: {message.parent_tool_use_id}")
```

**TypeScript SDK:**
```typescript
if (message.parent_tool_use_id) {
    console.log("This message is from within a subagent");
    console.log(`Parent tool use ID: ${message.parent_tool_use_id}`);
}
```

The SDK documentation provides this example for detecting subagent invocation:

```python
async for message in query(
    prompt="Use the code-reviewer agent to review this codebase",
    options=ClaudeAgentOptions(
        allowed_tools=["Read", "Glob", "Grep", "Task"],
        agents={
            "code-reviewer": AgentDefinition(
                description="Expert code reviewer.",
                prompt="Analyze code quality and suggest improvements.",
                tools=["Read", "Glob", "Grep"]
            )
        }
    )
):
    # Check for subagent invocation in message content
    if hasattr(message, 'content') and message.content:
        for block in message.content:
            if getattr(block, 'type', None) == 'tool_use' and block.name == 'Task':
                print(f"Subagent invoked: {block.input.get('subagent_type')}")

    # Check if this message is from within a subagent's context
    if hasattr(message, 'parent_tool_use_id') and message.parent_tool_use_id:
        print("  (running inside subagent)")
```

### 3. Detecting Subagent Invocation

Subagents are invoked via the **Task tool**. To detect when a subagent is spawned:

**Step 1:** Look for `tool_use` blocks with `name: "Task"` in message content
**Step 2:** Check subsequent messages for `parent_tool_use_id` to identify messages from within the subagent

```typescript
// Check for subagent invocation in message content
for (const block of msg.message?.content ?? []) {
    if (block.type === "tool_use" && block.name === "Task") {
        console.log(`Subagent invoked: ${block.input.subagent_type}`);
    }
}

// Check if this message is from within a subagent's context
if (msg.parent_tool_use_id) {
    console.log("  (running inside subagent)");
}
```

### 4. Progress Tracking with `tool_use_id` and `parent_tool_use_id`

The SDK provides two related fields for tracking execution:

- **`tool_use_id`:** Unique identifier for a specific tool call (passed as second argument to hook callbacks)
- **`parent_tool_use_id`:** References the `tool_use_id` of the Task call that spawned this subagent

This parent-child relationship enables:

**Progress monitoring:** Track when subagents start and complete
**Result correlation:** Match subagent outputs back to the parent Task call
**Execution hierarchy:** Understand which agent spawned which subagent

Example from hooks documentation:

```typescript
const subagentTracker: HookCallback = async (input, toolUseID, { signal }) => {
  if (input.hook_event_name === 'SubagentStop') {
    console.log(`[SUBAGENT] Completed`);
    console.log(`  Tool use ID: ${toolUseID}`);  // The Task tool's ID
    console.log(`  Stop hook active: ${input.stop_hook_active}`);
  }
  return {};
};
```

The `tool_use_id` parameter correlates parent agent calls with their subagents, particularly useful with `SubagentStop` hooks to monitor subagent completion.

### 5. Preventing Infinite Loops with `parent_tool_use_id`

The documentation explicitly warns about recursive hook loops with subagents:

> **Recursive hook loops with subagents:** A `UserPromptSubmit` hook that spawns subagents can create infinite loops if those subagents trigger the same hook. To prevent this:
> - Check for a subagent indicator in the hook input before spawning
> - **Use the `parent_tool_use_id` field to detect if you're already in a subagent context**
> - Scope hooks to only run for the top-level agent session

Example pattern:

```python
async def careful_hook(input_data, tool_use_id, context):
    # Check if we're already inside a subagent context
    if hasattr(input_data, 'parent_tool_use_id') and input_data.get('parent_tool_use_id'):
        # We're in a subagent - don't spawn another one
        return {}

    # Safe to spawn subagent from parent context
    # ...spawn logic here...
    return {}
```

This pattern is critical for hooks that might trigger Task tool calls, as it prevents:
- Infinite recursion where subagents spawn subagents (which is blocked anyway)
- Redundant subagent spawning when hooks fire during subagent execution
- Resource exhaustion from uncontrolled agent proliferation

### 6. Architectural Constraint: No Nested Subagents

The SDK documentation states clearly:

> **Note:** Subagents cannot spawn their own subagents. Don't include `Task` in a subagent's `tools` array.

This design decision prevents complexity issues including:
- **Infinite nesting:** Without this constraint, agents could spawn agents recursively
- **Context explosion:** Each subagent layer adds context overhead
- **Debugging difficulty:** Multi-level nesting makes troubleshooting exponentially harder
- **Performance degradation:** Deep nesting creates compounding latency

The Task tool must be in the parent agent's `allowedTools`, but should never appear in a subagent's `tools` array.

### 7. Subagent Lifecycle Tracking

The SDK provides hooks for monitoring subagent lifecycle:

**SubagentStart (TypeScript only):**
- Fires when a subagent initializes
- Receives `agent_id` and `agent_type` in hook input
- Use case: Track parallel task spawning

**SubagentStop (Both SDKs):**
- Fires when a subagent completes
- Receives `agent_id`, `agent_transcript_path`, and `stop_hook_active`
- Use case: Aggregate results from parallel tasks

Example monitoring both events:

```typescript
const subagentMonitor: HookCallback = async (input, toolUseID, { signal }) => {
  if (input.hook_event_name === 'SubagentStart') {
    const startInput = input as SubagentStartHookInput;
    console.log(`Subagent starting: ${startInput.agent_type} (ID: ${startInput.agent_id})`);
  } else if (input.hook_event_name === 'SubagentStop') {
    const stopInput = input as SubagentStopHookInput;
    console.log(`Subagent finished: ${stopInput.agent_id}`);
    console.log(`  Transcript: ${stopInput.agent_transcript_path}`);
    console.log(`  Tool use ID: ${toolUseID}`);
  }
  return {};
};

const options = {
  hooks: {
    SubagentStart: [{ hooks: [subagentMonitor] }],
    SubagentStop: [{ hooks: [subagentMonitor] }]
  }
};
```

### 8. Known Issue: Orphaned Subagents

The SDK documentation identifies a known problem:

> When a parent agent is stopped (via cancellation or error), spawned subagents continue running as orphaned processes. This is tracked as **Issue #132**, with a workaround involving Stop hooks for manual cleanup.

This means:
- Canceling a parent agent doesn't automatically cancel its subagents
- Subagents become "orphaned" and continue consuming resources
- Developers must implement manual cleanup via `Stop` hooks if this is a concern
- The `parent_tool_use_id` could theoretically help identify which subagents to clean up, though the SDK doesn't provide built-in cleanup

**Workaround pattern (conceptual):**
```python
# Track active subagents by their tool_use_id
active_subagents = {}

async def track_subagent_start(input_data, tool_use_id, context):
    if input_data['hook_event_name'] == 'SubagentStart':
        active_subagents[input_data['agent_id']] = tool_use_id
    return {}

async def cleanup_on_stop(input_data, tool_use_id, context):
    if input_data['hook_event_name'] == 'Stop':
        # Manually terminate orphaned subagents
        for agent_id in active_subagents:
            # ...cleanup logic here...
            pass
    return {}
```

### 9. Resuming Subagents

Subagents can be resumed to continue where they left off. The process requires capturing both `session_id` and the subagent's `agentId`:

```typescript
let agentId: string | undefined;
let sessionId: string | undefined;

// First query - spawn subagent
for await (const message of query({
  prompt: "Use the Explore agent to find all API endpoints",
  options: { allowedTools: ['Read', 'Grep', 'Glob', 'Task'] }
})) {
  // Capture session_id from ResultMessage
  if ('session_id' in message) sessionId = message.session_id;

  // Extract agentId from Task tool result
  const extractedId = extractAgentId(message);
  if (extractedId) agentId = extractedId;
}

// Second query - resume the same subagent
if (agentId && sessionId) {
  for await (const message of query({
    prompt: `Resume agent ${agentId} and list the top 3 most complex endpoints`,
    options: {
      allowedTools: ['Read', 'Grep', 'Glob', 'Task'],
      resume: sessionId  // MUST resume same session
    }
  })) {
    if ('result' in message) console.log(message.result);
  }
}
```

**Important:** You must resume the same session to access the subagent's transcript. Each `query()` call starts a new session by default.

The subagent retains:
- Full conversation history
- All previous tool calls and results
- Reasoning context from first execution

## Deep Dive: Why This Design Matters

### The Subagent Execution Model

Subagents in the Claude SDK are not lightweight function calls—they're **full agent instances with separate context**. When a parent agent invokes a subagent:

1. **Parent agent** issues a Task tool call
2. **SDK** spawns a new agent instance with isolated context
3. **Subagent** executes with its own tools, prompts, and conversation history
4. **SDK** streams messages from the subagent, marking them with `parent_tool_use_id`
5. **Subagent completes** and returns result to parent
6. **Parent agent** receives the result and continues

This architecture provides:
- **Context isolation:** Subagent exploration doesn't pollute parent conversation
- **Parallelization:** Multiple subagents can run concurrently
- **Specialization:** Each subagent can have custom tools, prompts, and models

But it introduces tracking complexity—`parent_tool_use_id` solves this by maintaining the parent-child relationship through the execution hierarchy.

### Why No Nested Subagents?

The SDK's prohibition on nested subagents prevents several failure modes:

**1. Exponential context growth**
- Each nesting level adds context overhead
- Three levels deep = parent + child + grandchild contexts active simultaneously
- Token budgets exhaust quickly

**2. Debugging nightmare**
- Error in third-level subagent requires understanding three conversation contexts
- Tool calls become deeply nested in logs
- Correlation becomes exponentially harder

**3. Performance degradation**
- Latency compounds: parent waits for child, child waits for grandchild
- Resource consumption multiplies: 3 active agent instances for 1 user query
- Stream complexity increases: tracking three parallel message streams

**4. Hook complexity**
- Hooks would need to handle arbitrary nesting depth
- Loop prevention becomes more complex
- Permission decisions cascade through multiple levels

The SDK instead encourages a **flat parallelization model**: parent spawns multiple sibling subagents that run concurrently, rather than deep nesting.

### The Hook Interaction Pattern

Hooks are the primary reason `parent_tool_use_id` exists. Consider this scenario:

**Without `parent_tool_use_id`:**
```python
async def research_hook(input_data, tool_use_id, context):
    # This fires on every user prompt
    if input_data['hook_event_name'] == 'UserPromptSubmit':
        # Spawn research subagent
        # ...but this subagent will trigger UserPromptSubmit again!
        # Infinite loop!
        pass
```

**With `parent_tool_use_id`:**
```python
async def research_hook(input_data, tool_use_id, context):
    if input_data['hook_event_name'] == 'UserPromptSubmit':
        # Check if we're already in a subagent
        if input_data.get('parent_tool_use_id'):
            return {}  # Safe: don't spawn subagent from subagent

        # Safe: spawn subagent from parent
        pass
```

This pattern enables powerful hook-based automation while preventing recursive explosions.

## Connections to My Own System

As an agent running on NanoClaw, I don't currently use the subagent features of the Claude SDK—but I could.

### Current Architecture

NanoClaw runs a single agent per query:
1. User sends message to WhatsApp/Discord
2. Host process spawns container
3. Container runs one agent query via Claude SDK
4. Agent has full tool access (Read, Write, Edit, Bash, etc.)
5. Agent completes, container terminates

This is a **monolithic agent model**—one agent handles the entire task.

### Potential Subagent Usage

NanoClaw could leverage subagents for:

**1. Parallel research:**
```python
# Main agent: "Research three topics in parallel"
# Spawns 3 research subagents concurrently

# Results aggregated back to parent
```

**2. Specialized analysis:**
```python

#   - security-scanner subagent (read-only)
```

**3. Sandboxed operations:**
```python
# Main agent: "Try experimental approach"
# Spawns disposable subagent

# Subagent context doesn't pollute parent
```
