# Claude SDK PreToolUse and PostToolUse Hooks - Timing Relationship with Streaming Events

**Research ID:** rq-1770837893000-hooks
**Completed:** 2026-02-12
**Tags:** sdk, hooks, streaming, agent-development

---

## Summary

PreToolUse and PostToolUse hooks operate as **synchronous gatekeeping checkpoints** in the Claude Agent SDK's execution pipeline. They execute **between streaming events**, creating deliberate pauses in the stream: PreToolUse fires after the model streams a `tool_use` content block but **before** the tool executes, and PostToolUse fires **after** tool execution completes but before the next model turn begins. During hook execution, streaming events pause while the SDK awaits the async hook callback's resolution.

## Key Findings

### 1. Hook Execution Points in the Event Timeline

The streaming + hook execution flow follows this sequence:

```
1. StreamEvent (message_start)
2. StreamEvent (content_block_start) - text block
3. StreamEvent (content_block_delta) - text chunks stream...
4. StreamEvent (content_block_stop)
5. StreamEvent (content_block_start) - tool_use block
6. StreamEvent (content_block_delta) - tool input JSON chunks stream...
7. StreamEvent (content_block_stop) - tool parameters fully received
   ⏸️  STREAMING PAUSES
8. 🪝 PreToolUse hook fires (async callback awaited)
9. 🪝 Hook callback returns decision (allow/deny/modify)
   ▶️  STREAMING RESUMES (if allowed)
10. ⚙️ Tool executes
11. ⏸️  STREAMING PAUSES
12. 🪝 PostToolUse hook fires (async callback awaited)
13. 🪝 Hook callback returns (optional system message, etc.)
    ▶️  STREAMING RESUMES
14. StreamEvent (message_delta)
15. StreamEvent (message_stop)
16. AssistantMessage - complete message with all content
17. ... next turn begins ...
```

**Critical insight:** Hooks execute **synchronously** (the SDK awaits their completion) even though the callbacks themselves are async functions. The stream does not continue until the hook callback resolves.

### 2. Correlation via Tool Use ID

Both PreToolUse and PostToolUse callbacks receive a `tool_use_id` parameter (string | null) that allows correlation of the same tool execution across both hooks:

```typescript
const logToolLifecycle: HookCallback = async (input, toolUseID, { signal }) => {
  if (input.hook_event_name === "PreToolUse") {
    console.log(`[${toolUseID}] Starting tool: ${input.tool_name}`);
  } else if (input.hook_event_name === "PostToolUse") {
    console.log(`[${toolUseID}] Completed tool: ${input.tool_name}`);
  }
  return {};
};
```

This enables tracking the complete lifecycle of individual tool executions across hook events.

### 3. Async Hook Callbacks with Timeout Context

Hook callbacks are async functions that receive three arguments:

1. **Input data** (dict/HookInput): Event details including `hook_event_name`, `tool_name`, `tool_input`, etc.
2. **Tool use ID** (str | None / string | null): For correlating Pre/Post events
3. **Context** (HookContext): In TypeScript, contains an `AbortSignal` for cancellation

The `signal` parameter enables proper async operation handling:

```typescript
const webhookNotifier: HookCallback = async (input, toolUseID, { signal }) => {
  try {
    await fetch("https://api.example.com/webhook", {
      method: "POST",
      body: JSON.stringify({ tool: input.tool_name }),
      signal // Cancels fetch if hook times out
    });
  } catch (error) {
    if (error.name === "AbortError") {
      console.log("Hook timeout - webhook cancelled");
    }
  }
  return {};
};
```

**Default timeout:** 60 seconds (configurable via `timeout` in `HookMatcher`)
**Maximum timeout:** 10 minutes (increased from 60s in v2.1.3)

### 4. Hook Data Availability by Phase

**PreToolUse receives:**
- `tool_name`: String identifying the tool
- `tool_input`: Object containing tool arguments (before execution)
- `session_id`, `transcript_path`, `cwd`: Context fields

**PostToolUse receives:**
- All PreToolUse fields, PLUS:
- `tool_response`: The actual result returned from tool execution

This asymmetry is critical: PreToolUse cannot see results (they don't exist yet), while PostToolUse has complete visibility into both input and output.

### 5. Impact on Streaming UX

Because hooks pause the stream, long-running hook callbacks create **perceived latency** in the user experience:

**Fast hooks (< 100ms):** Imperceptible pause, smooth streaming
**Slow hooks (> 1s):** Noticeable gap between tool call stream and execution
**Very slow hooks (> 5s):** Creates "frozen" feeling during wait

**Best practices for production:**
- Keep hook callbacks lightweight (< 100ms)
- Offload heavy operations to background tasks
- Use `signal` parameter to handle cancellation gracefully
- Set appropriate `timeout` values based on hook complexity

### 6. Permission Decision Flow and Blocking

Multiple hooks can affect the same tool execution. The SDK evaluates them in **priority order**:

1. **Deny** rules checked first → immediate blocking
2. **Ask** rules checked second → prompt user
3. **Allow** rules checked third → auto-approve
4. **Default to Ask** if nothing matches

**Critical behavior:** If ANY hook returns `permissionDecision: "deny"`, the tool is blocked regardless of other hooks returning `allow`. Deny decisions override everything.

Example blocking pattern:

```typescript
const protectSystemFiles: HookCallback = async (input, toolUseID, { signal }) => {
  const filePath = (input as PreToolUseHookInput).tool_input?.file_path as string;

  if (filePath?.startsWith("/etc") || filePath?.startsWith("/sys")) {
    return {
      systemMessage: "System directories are protected.",
      hookSpecificOutput: {
        hookEventName: input.hook_event_name,
        permissionDecision: "deny",
        permissionDecisionReason: "Cannot modify system directories"
      }
    };
  }
  return {};
};
```

## Deep Dive

### Hook Architecture Design

The Claude Agent SDK implements hooks as **interceptor middleware** in the execution pipeline. Unlike pure event observers (which passively watch), hooks have **active control** over execution flow:

- **Read access:** All hook events can read context (session, transcript, tool details)
- **Write access:** PreToolUse can modify tool inputs via `updatedInput`
- **Control flow:** Hooks can block (`deny`), prompt (`ask`), or allow (`allow`)
- **Injection:** Hooks can inject `systemMessage` visible to Claude in the conversation

This architecture enables use cases like:
- Security validation (block dangerous commands)
- Credential injection (add API keys to tool inputs)
- Audit logging (track all tool usage)
- Path redirection (sandbox file operations)

### Streaming Event Types Reference

When `include_partial_messages: true` (Python) or `includePartialMessages: true` (TypeScript), the SDK emits `StreamEvent` messages containing raw API events:

| Event Type | Description | Contains |
|------------|-------------|----------|
| `message_start` | Start of new message | Message metadata |
| `content_block_start` | Start of content block | Type: `text` or `tool_use` |
| `content_block_delta` | Incremental content | `text_delta` or `input_json_delta` |
| `content_block_stop` | End of content block | - |
| `message_delta` | Message-level updates | Stop reason, usage stats |
| `message_stop` | End of message | - |

**Tool use streaming:** Tool parameters stream as `input_json_delta` chunks. The model emits one complete key-value pair at a time, which can cause delays between streaming events while working on complex inputs.

### Hook Execution Issues (Known Bugs as of 2026)

Research uncovered several reported issues with hook execution:

1. **PostToolUse exit code 1 blocks execution** ([Issue #4809](https://github.com/anthropics/claude-code/issues/4809))
   - PostToolUse hooks with exit code 1 block Claude despite docs claiming "non-blocking"
   - Requires user intervention to unblock

2. **Hooks not firing** ([Issue #6305](https://github.com/anthropics/claude-code/issues/6305))
   - PreToolUse and PostToolUse sometimes configured correctly but never execute
   - Other hook types (Stop, UserPromptSubmit) work fine
   - No evidence of execution in logs

3. **Built-in tools don't trigger PostToolUse** ([Issue #101](https://github.com/anthropics/claude-agent-sdk-typescript/issues/101))
   - PostToolUse fires for custom MCP tools
   - Never fires for built-in tools like WebSearch, WebFetch
   - Issue appeared in SDK v0.1.63+

These issues suggest the hook system is still maturing and may have reliability concerns in production use cases.

### Matchers and Filtering

Matchers use **regex patterns** to target specific tools:

```python
options = ClaudeAgentOptions(
    hooks={
        "PreToolUse": [
            # Match file modification tools
            HookMatcher(matcher="Write|Edit|Delete", hooks=[file_security_hook]),

            # Match all MCP tools
            HookMatcher(matcher="^mcp__", hooks=[mcp_audit_hook]),

            # Match everything (no matcher)
            HookMatcher(hooks=[global_logger]),
        ]
    }
)
```

**Important limitation:** Matchers only match **tool names**, not arguments like file paths. To filter by file path, check `tool_input.file_path` inside the callback:

```typescript
const myHook: HookCallback = async (input, toolUseID, { signal }) => {
  const filePath = (input as PreToolUseHookInput).tool_input?.file_path as string;
  if (!filePath?.endsWith(".md")) return {}; // Skip non-markdown
  // Process markdown files...
};
```

### MCP Tool Naming Convention

MCP (Model Context Protocol) tools follow a specific naming pattern:

```
mcp__<server>__<action>
```

Examples:
- `mcp__playwright__browser_screenshot`
- `mcp__playwright__browser_click`
- `mcp__database__execute_query`

The server name comes from the key used in `mcpServers` configuration. Use matchers like `^mcp__playwright__` to target all tools from a specific MCP server.
